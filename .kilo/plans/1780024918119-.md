# Split Provider Logic: Kilo Gateway & GitHub Copilot

## Overview

Currently, both Kilo Gateway and GitHub Copilot providers share all logic through associative arrays in `lib/config.sh` and conditional checks scattered across `lib/api.sh` and `lib/slash.sh`. This plan splits the provider-specific logic into dedicated files while keeping shared infrastructure intact.

## Current Provider Differences

| Aspect | Kilo Gateway | GitHub Copilot |
|--------|-------------|----------------|
| Base URL | `https://gateway.kilocode.ai/v1` | `https://models.inference.ai.azure.com` |
| Default model | `openai/gpt-4o-mini` | `gpt-4o-mini` |
| Models endpoint | Separate: `https://api.kilo.ai/api/gateway/models` | Base URL + `/models` |
| Has /models at base | `false` | `true` |
| API key env var | `KILO_API_KEY` | `GITHUB_TOKEN` |
| Context field name | `context_length` | `max_context` |

## File Structure Changes

```
lib/
├── config.sh              # Keep: shared config loading, dispatch to provider
├── api.sh                 # Refactor: use provider dispatch functions
├── slash.sh               # Refactor: use provider dispatch for model parsing
├── providers/             # NEW: provider-specific implementations
│   ├── provider.sh        # NEW: provider interface/dispatch layer
│   ├── kilo.sh            # NEW: Kilo Gateway provider
│   └── github.sh          # NEW: GitHub Copilot provider
```

## Implementation Steps

### Step 1: Create Provider Directory & Interface

**File: `lib/providers/provider.sh`**

Create the provider dispatch layer that:
- Defines the provider interface (functions all providers must implement)
- Loads the appropriate provider file based on `BAISH_PROVIDER`
- Exposes unified provider functions that dispatch to the loaded implementation

Interface functions to be implemented by each provider:
- `provider_init()` - Provider-specific initialization
- `provider_api_key()` - Returns the API key from provider-specific env var
- `provider_base_url()` - Returns the API base URL
- `provider_default_model()` - Returns the default model name
- `provider_models_url()` - Returns the /models endpoint URL (or empty if not supported)
- `provider_has_models_endpoint()` - Returns "true" or "false"
- `provider_context_field()` - Returns the JSON field name for context window (`context_length` or `max_context`)
- `provider_parse_models(json)` - Parses model list from API response, outputs model names

### Step 2: Create Kilo Provider

**File: `lib/providers/kilo.sh`**

```bash
# Kilo Gateway provider implementation
# Implements the provider interface for https://gateway.kilocode.ai

provider_init() {
    # Kilo-specific initialization (if any)
    :
}

provider_api_key() {
    echo "${KILO_API_KEY:-}"
}

provider_base_url() {
    echo "https://gateway.kilocode.ai/v1"
}

provider_default_model() {
    echo "openai/gpt-4o-mini"
}

provider_models_url() {
    echo "https://api.kilo.ai/api/gateway/models"
}

provider_has_models_endpoint() {
    echo "false"  # Not at base URL
}

provider_context_field() {
    echo "context_length"
}

provider_parse_models() {
    local json="$1"
    # Kilo returns OpenAI-compatible format: {"data":[{"id":"model",...}...]}
    echo "$json" | jq -r '.data[].id // empty' 2>/dev/null
}
```

### Step 3: Create GitHub Provider

**File: `lib/providers/github.sh`**

```bash
# GitHub Copilot provider implementation
# Implements the provider interface for https://models.inference.ai.azure.com

provider_init() {
    # GitHub-specific initialization (if any)
    :
}

provider_api_key() {
    echo "${GITHUB_TOKEN:-}"
}

provider_base_url() {
    echo "https://models.inference.ai.azure.com"
}

provider_default_model() {
    echo "gpt-4o-mini"
}

provider_models_url() {
    # GitHub supports /models at the base URL
    echo ""
}

provider_has_models_endpoint() {
    echo "true"
}

provider_context_field() {
    echo "max_context"
}

provider_parse_models() {
    local json="$1"
    # GitHub returns flat array: [{"name":"model","task":"chat-completion"}...]
    # .id is an Azure URI, not usable as model name
    echo "$json" | jq -r '.[] | select(.task == "chat-completion") | .name // empty' 2>/dev/null
}
```

### Step 4: Update Config Loading

**File: `lib/config.sh`**

Changes:
1. Remove the four associative arrays (`_PROVIDER_BASE_URLS`, `_PROVIDER_DEFAULT_MODELS`, `_PROVIDER_HAS_MODELS_ENDPOINT`, `_PROVIDER_MODELS_URL`)
2. Remove provider resolution logic from `config_resolve_provider()` - delegate to provider interface
3. Source the provider dispatch layer after loading config

Updated `config_resolve_provider()`:
```bash
config_resolve_provider() {
    # Source the provider interface
    source "${_BAISH_DIR}/lib/providers/provider.sh"
    
    # Load provider-specific module
    provider_load "$BAISH_PROVIDER"
    
    # API key fallback (only if not explicitly set)
    if [[ -z "$BAISH_API_KEY" ]]; then
        BAISH_API_KEY="$(provider_api_key)"
    fi
    
    # Base URL auto-resolution
    if [[ -z "$BAISH_BASE_URL" ]]; then
        BAISH_BASE_URL="$(provider_base_url)"
    fi
    
    # Default model if still empty
    if [[ -z "$BAISH_MODEL" ]]; then
        BAISH_MODEL="$(provider_default_model)"
    fi
}
```

### Step 5: Refactor API Functions

Updated `api_models_url()`:

api_models_url() {
    local url
    url="$(provider_models_url)"
    if [[ -n "$url" ]]; then
        echo "$url"
    else
        echo "${BAISH_BASE_URL}/models"
    fi
}

## Risks & Considerations

1. **Backwards compatibility**: The `BAISH_PROVIDER`, `BAISH_MODEL`, `BAISH_API_KEY`, `BAISH_BASE_URL` environment variables and config file keys remain unchanged
2. **Error handling**: Provider loading should fail gracefully if an unknown provider is specified
3. **Future providers**: Adding a new provider only requires creating a new file in `lib/providers/` with the interface functions
