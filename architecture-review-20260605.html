Collapse Display + Error Output into one User Output module

Strong in-process

lib/agent/display.sh · lib/agent/errors.sh · lib/agent/config.sh (colors+icons)

Before — 3 shallow output slabs

display.sh — banner, spinner, info/error print
errors.sh — error codes, auth/context guidance
config.sh — ANSI colors, Unicode icons
run-loop.sh commands.sh bin/baish
each caller imports 3 files for output

After — 1 deep output seam

User Output

baish_output_banner

baish_output_info / error / thinking

baish_output_context_overflow_help

baish_output_auth_failure

run-loop.sh commands.sh bin/baish

Problem

User-facing output is split across three files: display.sh (banner/spinner/response printing), errors.sh (error guidance messages), and icon/color constants in config.sh. Every caller sources all three. Each is shallow — interface nearly matches implementation.

Solution

Merge display formatting + error guidance into one User Output module. Keep color/icon constants internal. Callers source a single module for all human-readable output.

Wins

  • Locality: all output formatting concentrates in one place
  • Leverage: one import for callers, not three
  • Interface shrinks: 3 sources → 1 source directive
  • Tests hit one output seam instead of testing display + error helpers separately

Problem

The provider auth→model→state lifecycle is duplicated across baish_startup (70 lines in bin/baish), /connect, /provider, and /model in commands.sh. Each copy is slightly different — different auth checks, different state write timing. Fixing a bug (e.g., auth expiry detection) requires patching every copy.

Solution

Extract a single Provider Activation module with one entry point: baish_provider_activate [--pick-provider] [--pick-model]. It handles the full lifecycle: restore state, check auth, interactively select if needed, persist. The four call sites shrink to one-liners.

Wins

  • Locality: auth expiry bug fixed once, not in 3+ places
  • Leverage: startup, /connect, /provider, /model all cross one seam
  • Interface: one function sig replaces 70+ lines of inline logic
  • Testable: the activation module can be tested with mock providers, unlike the current inline startup

Remove provider format knowledge from Session module

Worth exploring in-process

lib/agent/session.sh · lib/providers/copilot.sh · lib/providers/kilo.sh

Before — Session knows OpenAI tool call format

session.sh: "if has('function') then . else wrap"
↳ knows about {id, type: "function", function: {name, args}}
session.sh: "tool_calls normalization idempotent"
↳ format leakage across the seam

After — each Provider owns its format

session.sh: stores messages as-is
↳ agnostic — no format knowledge
copilot.sh / kilo.sh: return normalized format
↳ each Provider adapter normalizes its own output

Problem

baish_session_append_assistant_response contains a jq normalizer that rewrites tool calls from the internal flat format ({id, name, arguments}) to OpenAI’s wrapped format. This is a Provider adapter concern leaking into the Session module’s seam.

Solution

Require each Provider’s chat function to return tool calls already in OpenAI format. The Session module stores whatever it receives without transformation. Copilot and Kilo already produce this format; the normalizer is only needed because mock returns the flat form. Make mock return the wrapped form.

Wins

  • Cleaner seam: session doesn’t know about provider-specific formats
  • One adapter = hypothetical seam. Two adapters = real seam. The normalizer currently masks the real interface.
  • Mock provider tests get simpler (the existing tests already work when normalizing)

Separate debug logging from error handling

Speculative in-process

lib/agent/errors.sh · lib/agent/config.sh

Before — errors.sh carries debug baggage

Interface
error codes + debug_http + debug_tool + debug_state + 4 print funcs
Impl
printf + jq wrappers
shallow: interface nearly as wide as implementation

After — error handling deepens, debug gets its own home

Interface
error codes + handle_provider_error
Impl
error detection + guidance messages + state machine
deep: small interface, rich implementation

Problem

errors.sh defines error type constants and error handling logic — but also contains three specialized debug logging functions (baish_debug_http, baish_debug_tool, baish_debug_state). Meanwhile, the core baish_debug lives in config.sh. Debug logging is scattered across two files and muddies the error handling interface.

Solution

Move baish_debug_http, baish_debug_tool, baish_debug_state into config.sh alongside the existing baish_debug. errors.sh then only exports error constants and baish_handle_provider_error. One concern per file.

Wins

  • Locality: all debug logging lives in one file
  • Interface clarity: errors.sh exports only error handling — no debug noise
  • Low risk: pure function moves, no behavior changes

Top recommendation

Extract Provider Activation lifecycle

This candidate has the highest friction-to-effort ratio. The provider auth→model→state sequence is duplicated across 3+ call sites today, and every future slash command that touches provider state will add another. Concentrating this into one module creates immediate locality wins and makes the currently untested baish_startup testable through its new seam. Tackle this first; the Output collapse (Candidate 1) is a natural follow-up since the activation module will be one of the callers that benefits from a single output import.
